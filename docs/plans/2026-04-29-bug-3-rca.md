## TL;DR

Audit 2's framing is **partially wrong**. Bug 3 is real and reproduces, but
the per-stage breakdown in Audit 2 was incorrect (it claimed `[B+P] FAIL`,
which is false — `[B+P]` PASSES). The actual rejection is a
**Precedence × non-Precedence-annotation-semiring interaction** triggered
when two `AssignmentExpression` derivations within the same `ForStatement`
header advance a chart item where the `Precedence` slot's accumulated
value (`level=101 assoc=undef`) collides with another `level=101` value
during a chart-complete multiply. The interaction only manifests when
Precedence is paired with at least one other annotation semiring
(TypeInference or Structural) — under just `[Boolean, Precedence]` the
chart-cell merging is different and the offending multiply does not occur.

The right-fix is **not** in the `for-step` header treatment per se. The
underlying defect is that `Chalk::Bootstrap::Semiring::Precedence::_complete_prec`
returns the wrong `assoc` for `AssignmentExpression`. There is **dead
code** at lines 414–417 that intends to set `assoc='right'` for
`AssignmentExpression`, but it is unreachable because the
`$EXPR_LEVELS->{$rule_name}` clause at lines 395–401 catches
`AssignmentExpression` first and returns `_intern(true, 101, undef, false)`
— `assoc=undef`, which `_prec_multiply` defaults to `'left'`. That makes
`AssignmentExpression` look left-associative for same-level multiply
purposes, which causes line 306–308 (`parent_assoc eq 'left' && !$right_is_op`
→ reject) to misfire whenever two `AssignmentExpression`s meet in a
chart-complete multiply.

Categorization: **Category C** (interaction). The Precedence semiring is
buggy on its own (assoc=undef for AssignmentExpression), but the bug
only manifests in interaction with a second annotation semiring whose
presence changes the chart cell wrap-Context identity and exposes a
multiply path that B+P alone short-circuits.

## Root cause

### Code path of rejection

Captured in `/tmp/bug3-trace5.pl` (caller-stack instrumentation on a
TypeInference subclass that wraps `Precedence::multiply` to log
zero-returns):

```
PREC_REJ stack:
  lib/Chalk/Bootstrap/Semiring/FilterComposite.pm:163 StackTracePrec::multiply
  lib/Chalk/Bootstrap/Earley.pm:1337 Chalk::Bootstrap::Semiring::FilterComposite::multiply
  lib/Chalk/Bootstrap/Earley.pm:609 Chalk::Bootstrap::Earley::_complete
  lib/Chalk/Bootstrap/Earley.pm:1020 Chalk::Bootstrap::Earley::_run_parse
```

`Earley.pm:1337` is the chart-complete multiply: a waiting item is
advanced by multiplying with a completed sub-rule. Slot values at the
rejection point (from `/tmp/bug3-trace4.pl`):

```
L = (level=101  op=undef  assoc=undef  isop=F)
R = (event=mul rule=? alt=? matched=<ref> prec=(level=101 op=undef))
```

Both sides at `level=101` (AssignmentExpression). Left has `assoc=undef`.
In `_prec_multiply` (line 271): `my $parent_assoc = $left->{assoc} // 'left';`
→ defaults to `'left'`. Then line 305–308:

```perl
my $right_is_op = $right->{is_operator};
if ($parent_assoc eq 'left' && !$right_is_op) {
    return $self->zero();
}
```

Right is not an operator (`event=mul` chart-complete, not a BinaryOp/AssignOp
scan). Same level + (defaulted) left-assoc + non-operator right → reject.

### The latent defect: dead code in `_complete_prec`

`Chalk::Bootstrap::Semiring::Precedence::_complete_prec`
(`lib/Chalk/Bootstrap/Semiring/Precedence.pm:381–466`) handles
rule-completion. It contains two branches that both target
`AssignmentExpression`:

```perl
# Line 395–401  (first match wins)
if (defined($EXPR_LEVELS->{$rule_name})) {
    my $expr_level = $EXPR_LEVELS->{$rule_name};
    if ($expr_level < 0 && defined($value->{level}) && $value->{level} >= 0) {
        return $self->zero();
    }
    return _intern(true, $expr_level, undef, false);     # <-- assoc=undef
}

# Line 414–417  (UNREACHABLE for AssignmentExpression)
if ($rule_name eq 'AssignmentExpression') {
    return _intern(true, $EXPR_LEVELS->{AssignmentExpression}, 'right', false, $value->{op});
}
```

Because `$EXPR_LEVELS->{AssignmentExpression}` is defined (`= 101`), the
first clause matches and returns with `assoc=undef`. The second clause
is dead code. It exists but never runs.

The dead clause indicates **the original author intended
AssignmentExpression to complete with `assoc='right'`**. Operator
associativity matters for the same-level reject in `_prec_multiply`:
right-associative same-level RHS is permitted, left-associative same-level
RHS is rejected. With `assoc=undef`, `// 'left'` defaults left, and
the rejection misfires for any `AssignmentExpression × AssignmentExpression`
multiply path.

### Why does this only fire with `[B, P, X]` and not `[B, P]`?

Hypothesis (not fully instrumented, but consistent with all observed
behavior):

When `FilterComposite` has only `[Boolean, Precedence]`, the wrapping
Context produced by `_wrap_sa_result` carries only `boolean` and
`precedence` annotation slots. Two derivations of the same chart cell
produce wrapping Contexts that compare-equal under `_filter_compare` for
the precedence slot when their accumulated levels match (because
`_intern` is hash-consed: identical `(valid, level, assoc, is_operator)`
tuples share refaddrs). Earley's `add` short-circuits or the chart's
multiply chain reaches the `for`-header completion via a single path.

When TypeInference (or Structural) is added, the wrapping Context now
also carries `type` (or `structural`) annotations. These differ between
the two derivations of the same cell (different TI types produced by
different sub-derivations). `_filter_compare` skips TI for the type slot
(lines 234) but the wrapping Context's overall identity differs. The
second derivation isn't merged silently, and Earley extends *both* via
chart-complete multiplies. One of those multiplies hits the
`level=101 × level=101` left-assoc rejection.

This is consistent with `[B, T]` PASSING but `[B, P, T]` FAILING:
T alone doesn't reject anything in this path; but T's presence beside P
exposes a P-multiply that B+P alone never executed.

### Confirmed mechanism

The mechanism is a left-associative same-level rejection misfiring
because `AssignmentExpression`'s associativity is dropped during
`_complete_prec`. The unreachable `assoc='right'` clause confirms this
was unintended.

## Categorization

**Category C: Interaction.**

- Precedence has a latent bug (assoc=undef for AssignmentExpression
  via dead-code) that does not surface on its own under standard test
  coverage.
- Adding any non-Precedence annotation semiring exposes the bug because
  the chart's wrap-Context identity differs and chart-complete multiply
  paths are taken that B+P alone collapses.

This is a Category-A bug (Precedence logic is wrong) that **manifests
as Category C** (only in interaction). The fix is in Precedence, not
in the grammar's `ForStatement` rule.

## Proposed remediation

### Primary fix

In `lib/Chalk/Bootstrap/Semiring/Precedence.pm`, change the
`_complete_prec` handler so that `AssignmentExpression` returns with
`assoc='right'`. There are two equivalent shapes:

**Option 1: extend the `$EXPR_LEVELS` clause to know per-rule assoc.**

Convert `$EXPR_LEVELS` from `{ rule => level }` to
`{ rule => [level, assoc] }`. At line 395–401, return the per-rule assoc:

```perl
my $EXPR_LEVELS = {
    PostfixExpression    => [-2,  undef],
    UnaryExpression      => [-1,  undef],
    TernaryExpression    => [100, 'right'],   # ?: is right-assoc
    AssignmentExpression => [101, 'right'],
};

# in _complete_prec
if (my $info = $EXPR_LEVELS->{$rule_name}) {
    my ($expr_level, $expr_assoc) = $info->@*;
    if ($expr_level < 0 && defined($value->{level}) && $value->{level} >= 0) {
        return $self->zero();
    }
    return _intern(true, $expr_level, $expr_assoc, false);
}
```

Delete the unreachable `if ($rule_name eq 'AssignmentExpression') {…}`
clause (lines 414–417). Optionally lift the `op` carry: line 416 was
trying to preserve `$value->{op}` — verify whether downstream multiplies
need it. (Likely not: `op` is debug text; the trace shows
`prec=(level=101 op=undef)` already.)

**Option 2: reorder `_complete_prec` so `AssignmentExpression` is matched
first.**

Move the `if ($rule_name eq 'AssignmentExpression')` clause above the
`$EXPR_LEVELS` clause at line 395. This is a smaller textual change but
leaves the dead-code shape latent for other rules.

**Recommendation: Option 1.** It removes the dead code, makes the
intent explicit (per-rule associativity), and admits a clean place to
add `TernaryExpression` associativity should that ever matter.

### What does NOT change

- `Chalk::Bootstrap::Semiring::TypeInference` — no changes.
- `Chalk::Bootstrap::Semiring::FilterComposite` — no changes.
- `docs/chalk-bootstrap.bnf` `ForStatement` rule — no changes. The
  grammar is correct.
- `lib/Chalk/Bootstrap/Optimizer/StructPromotion.pm` — no changes.
- `lib/Chalk/Bootstrap/Perl/Target/C.pm:107` — no changes.
- `_scan_multiply` AssignOp branch — no changes (Audit 2's hypothesis
  about this path is wrong; it doesn't fire).

### Why not "treat ForStatement as RESETS"?

Audit 2 § "Why this is incorrect behavior" recommended "the Precedence
semiring's accumulated state should reset between the three
semicolon-separated parts" via adding `ForStatement` to the `$RESETS`
table. **This is incorrect**:

1. The rejection isn't between successive ForStatement slots — the
   precedence-level accumulation works correctly across `;` boundaries
   in `[B, P]` alone. Adding ForStatement to RESETS would over-reset:
   the outer enclosing context (e.g., `ExpressionStatement`) still
   needs precedence info from the inner Block.
2. `$RESETS` is a `_complete_prec` table that fires on rule-completion,
   not slot-by-slot. By the time `ForStatement` completes, all three
   slots are already parsed; resetting then is too late to influence the
   in-flight multiply that's rejecting.
3. The dead-code in `_complete_prec` is the actual defect. Fixing
   associativity addresses the root cause; the RESETS approach masks it.

## Side effects

### Cross-effects with recent fixes

- **Fix A** (`ac0e66ce`) added a `PostfixDeref` bracket-reset to
  Precedence's `_scan_multiply`. That change is in the scan path; this
  fix is in the complete path. They don't interact, but Option 1
  (per-rule assoc table) gives Fix A's pattern a more consistent home
  if `PostfixDeref`'s level/assoc ever needs to round-trip through
  `_complete_prec`.
- **A3 walker hygiene** (`fc4524ef`, `45d8c131`) is TI-side. No
  cross-effect.
- **Bug 5** (`d7432d43`) is TI walker depth. No cross-effect.

### Effect on `TernaryExpression`

`TernaryExpression` shares the `$EXPR_LEVELS` mechanism. It currently
returns `assoc=undef` (also defaulting to `'left'`). Perl's `?:` is
right-associative. The same multiply rejection could in principle fire
for two same-level Ternary expressions adjacent in a chart-complete
path. **Probe before fixing**: `/tmp/bug3-probe-ternary.pl` (proposed,
not yet written) should test `for (my $a = $x ? 1 : 2; ; $b = $y ? 3 : 4)`
patterns before and after the fix.

### Effect on `PostfixExpression` / `UnaryExpression`

These have negative levels and don't go through the same-level reject
in `_prec_multiply`. Setting `assoc=undef` for them is semantically
fine. Option 1 keeps `assoc=undef` for negative-level rules, so this
fix doesn't change their behavior.

### Effect on the `_scan_multiply` AssignOp path

`_scan_multiply` at line 167–188 has its own AssignOp handling that
uses `'right'` assoc explicitly. The `is_operator=true` interplay there
is checked separately. The fix targets `_complete_prec`, not
`_scan_multiply`, so the AssignOp scan path is unaffected.

### Effect on chained-assignment parsing

`$a = $b = $c` has level=101 nested. After the fix:
- Inner `$b = $c` completes as `_intern(true, 101, 'right', false)`.
- Multiplied with the outer accumulated `=` AssignOp scan
  (level=101, is_operator=true).
- `_prec_multiply` line 246–250: `if ($left_level == $op_level && $op_assoc eq 'right')` → reject.
- Wait — that path triggers only when right is an operator. Inner
  AssignmentExpression isn't an operator. So we hit lines 305–319.
  With `parent_assoc='right'` (from the fix) and `$right->{is_operator}`
  off → no reject.

Net: chained right-assoc assignments continue to parse correctly.

### Effect on `(my $x = $y) //= 1` (the comment at line 165)

The comment claims this should reject. With the fix:
- Inner `(my $x = $y)` is a ParenExpr → `$RESETS` → `one()` → no
  precedence carry.
- Then `//=` AssignOp scans. Left is `one()`, level=undef. AssignOp
  scan path at line 167–188: `$existing->{level}` undefined → first
  conditional false → second conditional false → falls through to
  `_intern(true, 101, 'right', true, '//=')`. No reject.
- So `(my $x = $y) //= 1` does NOT reject under the current code.
  The line 162–165 comment may already be stale, predating ParenExpr's
  RESETS treatment. **Document but do not fix in this RCA.**

## Latent rejection risk

### Probe other Expression-level rules

`$EXPR_LEVELS` covers four rules. Before the fix:
- `PostfixExpression` (level=-2): negative, never hits same-level reject.
- `UnaryExpression` (level=-1): negative, never hits same-level reject.
- `TernaryExpression` (level=100): assoc=undef (default 'left'). Perl's
  `?:` is right-assoc. **Same latent issue.** Probe needed: do any
  paths combine two `TernaryExpression`s at the same chart level?
- `AssignmentExpression` (level=101): the bug subject.

### Probe `for ($a ?: $b ; ; $c ?: $d)`

If `Ternary` has the same latent issue, `for` headers with two
`TernaryExpression`s should fail similarly. **Implementer should run
this probe before and after the fix.**

### Probe other `Precedence × X` interactions

The mechanism (chart-complete multiply that B+P alone collapses,
exposed by adding a third annotation semiring) may exist for other
combinations. Likely candidates:
- `Precedence × Structural` — same shape per the probe table. Fixing
  AssignmentExpression's assoc should also fix the B+P+S failure
  because the underlying multiply rejection is identical.
- `Precedence × SemanticAction` — SA is the last semiring; FC's
  `_filter_compare` doesn't iterate SA. SA's effect is on the wrap
  Context's `focus`. Probe needed: does any SA-side variation expose
  Precedence multiplies that B+P collapses?

### Probe nested `for` in `for`

`for (my $i = 0; ; for (my $j = 0; ; $i += 2)) { ... }` is unusual but
syntactically possible (Perl `for` accepts any Expression slot). Likely
also fails today. Should be covered by the fix because the underlying
rejection is the same.
