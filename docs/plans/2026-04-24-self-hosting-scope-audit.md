# Self-Hosting Scope Audit: Ambiguity/Grammar Questions vs `lib/`

Date: 2026-04-24
Companion to: `2026-04-24-toke-sweep-undocumented-ambiguity.md`

## Overview

This audit walks each of the 10 open questions raised by the toke.c sweep and
determines whether `lib/` (the Chalk self-hosted source) actually exercises the
grammar gap. The result is a narrower list of what must be resolved for
self-hosting, separated from aspirational scope expansion for a general Perl
compiler.

Scope: 150 `.pm` files in `lib/`. Q1 (POD) and Q3 (heredocs) were already
resolved via preprocessor hooks and are excluded.

Summary:

- **In scope for self-hosting (4)**: Q2 (quote-like delimiters), Q4 (string
  interpolation internals), Q6 (slices — only the single-Expression form
  `@arr[$x..$y]`, not the list form), Q9 (`-X` file tests).
- **Partial / easy to cover (1)**: Q7 — every attribute arg used in `lib/`
  fits the current grammar (QualifiedIdentifier), but `:param(name)` is part
  of the same category and needs verification that it actually parses today.
- **Out of scope (5)**: Q5 (FUNC/LSTOP), Q8 (`\&foo`), Q10 (version strings
  in expressions), Q11 modulo-vs-hash-sigil and readline-vs-less-than (the
  modulo case is used but is trivially unambiguous in `lib/`; the `<$fh>`
  case IS used but is one site), Q20 (formats).

Other gaps discovered while auditing (not on the original question list):

- `local $/;` — the grammar's `ScalarVariable` regex does not match
  punctuation variables (`$!`, `$@`, `$/`, `$,`).
- `eval "string"` — Chalk is documented to use `try/catch` only, but
  `DepChaser.pm` contains a `eval "$generated; 1"` call on real code.
- `<$fh>` readline (Q11) — one real site in `DepChaser.pm`.

## Methodology

For each question, I ran targeted `ag` searches against `lib/` looking for
the specific syntactic form the question describes. I then checked whether
the form actually used in `lib/` falls inside the grammar's admitted shape
(cross-referenced against `docs/chalk-bootstrap.bnf`).

A question is "in scope" when `lib/` contains at least one site that the
current grammar cannot parse. A question is "out of scope" when either no
occurrences exist, or the occurrences are restricted forms that the grammar
already handles.

Evidence is reported as `file:line` citations, with quoted source.

## Per-question analysis

### Q2: Quote-like operator delimiters

**Grammar gap.** Grammar admits only: `q{}`, `q[]`, `qq{}`, `qq[]`, `m//`,
`m{}`, `qr//`, `s///`, `s{}{}`, `qw()`. Not admitted: `qw{}`, `qw/.../`,
`qr{}`, `q()`, `q//`, `tr///`, `y///`, etc.

**Usage in lib/.**

- `qw(...)` — 60 occurrences across 10 files. Supported.
- `qw{}`, `qw[]`, `qw//` — 0 occurrences. Not used.
- `q(...)` — 1 occurrence: `lib/Chalk/Bootstrap/BNF/Target/C.pm:593`
  `return q(/* ABOUTME: Shared runtime header ... */)`.
- `q{...}` — 1 occurrence: `lib/Chalk/Bootstrap/BNF/Target/XS/AST/Preamble.pm:11`
  `return q{/* Generated by Chalk::Bootstrap ... */}`.
- `q[...]` — 2 occurrences in `lib/Chalk/Bootstrap/BNF/Target/Perl.pm` (lines
  21, 117): `return q[# ABOUTME: Generated BNF meta-grammar ...]`.
- `qq{...}` — many occurrences (e.g. `lib/Chalk/Bootstrap/ConciseTree/Actions.pm:493`
  `type_info => qq{PV "$ident_text"}`). Supported.
- `qq(...)` — e.g. `lib/Chalk/Bootstrap/BNF/Target/C.pm:27` `return qq("$s");`
  and `lib/Chalk/Bootstrap/BNF/Target/C.pm:141` `my $c_body = qq(#include ...)`.
- `qq[...]` — e.g. `lib/Chalk/Bootstrap/BNF/Target/XS.pm:262` `return qq[# Generated ...]`.
- `qr//` — the only `qr` form used (several sites in `Chalk/Bootstrap/Earley.pm`,
  `Chalk/Bootstrap/Perl/Target/EmitHelpers.pm`). Supported.
- `qr{}`, `qr[]`, `qr()` — 0 occurrences. Not used.
- `tr///`, `y///` — 0 occurrences. Not used.
- `m{...}` — 23 occurrences. Supported by grammar.
- `s{...}{...}...` — 41 occurrences. Supported.

**Does Chalk currently handle it?** For `q[...]`, `q(...)`, `qq[...]`,
`qq(...)`, NO. The StringLiteral rule only supports `q{}`, `q[]`, `qq{}`,
`qq[]`. Wait — the grammar does support `q[...]` (see line 298) and `qq[...]`
(line 300). But `q(...)` and `qq(...)` with parens are NOT in the grammar.

Re-reading the grammar:

```
StringLiteral ::= /'(?:[^'\\]|\\.)*'/
    | /"(?:[^"\\]|\\.)*"/
    | /q\s*\{(?:[^}\\]|\\.)*\}/
    | /q\s*\[(?:[^\]\\]|\\.)*\]/
    | /qq\s*\{(?:[^}\\]|\\.)*\}/
    | /qq\s*\[(?:[^\]\\]|\\.)*\]/
    | /`(?:[^`\\]|\\.)*`/ ;
```

So `q[...]` IS supported, `qq[...]` IS supported. `q(...)` and `qq(...)`
are NOT supported.

**In scope for self-hosting?** YES.
- Two sites use `q(...)` / `qq(...)` forms the grammar rejects:
  - `lib/Chalk/Bootstrap/BNF/Target/C.pm:593` (`q(...)`)
  - `lib/Chalk/Bootstrap/BNF/Target/C.pm:27` and `141` (`qq(...)`)
- Work needed: add `q\s*\(...\)` and `qq\s*\(...\)` to `StringLiteral`
  alternatives. Balanced-paren matching makes this slightly more involved
  than the `{}` / `[]` forms (Perl requires paired-paren handling for
  `q(ab(cd)ef)`; Chalk's current `{}` regex doesn't handle nesting either,
  so the same "outer-only" semantics would be acceptable for the self-hosted
  code, since all actual uses contain no inner parens).

The alternative is trivial: rewrite the three sites in `lib/` to use `q{}`
/ `qq{}`. That's a 3-line edit. Given the small footprint, rewriting `lib/`
is the lower-risk option.

### Q4: Interpolated string internals

**Grammar gap.** Strings are treated as opaque regex-matched tokens; grammar
doesn't analyze `$foo[0]`, `${foo}bar`, `$foo{k}`, `${\ expr}`, or
`$obj->method()` inside strings.

**Usage in lib/.**

- `${name}` identifier-in-braces interpolation — heavy use:
  - `lib/Chalk/Bootstrap/Optimizer/StructPromotion.pm` — multiple uses including
    `"${var_prefix}::${var_name}"`, `"${class_name}::${method_name}"`,
    `"struct_${schema_counter}_t"`.
  - `lib/Chalk/Bootstrap/Desugar.pm:57` `"${base}_$suffix{$quant}"`
  - `lib/Chalk/Bootstrap/Perl/Target/Perl.pm` — several sites.
  - `lib/Chalk/Bootstrap/Perl/Target/C.pm` — many sites.

- `${\ expr}` block-form interpolation (dereferenced SCALAR ref, idiomatic way
  to interpolate expressions) — real usage:
  - `lib/Chalk/Bootstrap/Optimizer/StructPromotion.pm:300,327,369,403`:
    `"${var_prefix}::${\$value->value()}"`,
    `"${var_prefix}::${\$arg->value()}"`, etc.
  - `lib/Chalk/Bootstrap/Perl/Target/EmitHelpers.pm:1501,1540,1599,1640`:
    e.g. `"get_sv(\"${\$self->module_name()}::$inner\", GV_ADD)"`.
  - `lib/Chalk/Bootstrap/Perl/Target/C.pm:141,179`:
    `"${\  $self->_get_current_slug()}_${name}"`, `"PERL_UNUSED_ARG(${\($p =~ s/^SV \*//r)});"`.

- `$arr[N]` / `$hash{k}` inside strings — not widely used as interpolation
  (hits were mostly code outside strings). No real examples of interpolated
  subscripts found.

**Does Chalk currently handle it?** Partial. The grammar defines StringLiteral
as an opaque token:

```
/"(?:[^"\\]|\\.)*"/
```

That regex successfully matches any of the above constructs as a whole token
(because `"` is only terminated by an unescaped `"`). Interpolation happens
at runtime in the generated Perl, and the backends emit the string text
verbatim. The **compile-time grammar** therefore accepts these strings.

What the grammar does not do: structurally analyze the interpolated
sub-expression. That does not matter for self-hosting as long as the runtime
Perl is the target (Chalk emits a `"..."` containing the same text, and Perl
interprets the interpolation). It WOULD matter for a non-Perl backend (XS/C
bytecode emission of interpolated strings needs to analyze the internals).

**In scope for self-hosting?** PARTIAL — YES for the XS/C backends, NO for
Perl output.

For Perl-only self-hosting, Q4 is not blocking: the grammar accepts these
strings and the backend passes them through.

For XS/C backends, the Interpolate IR node (`lib/Chalk/IR/Node/Interpolate.pm`
exists) needs to know what the `${\ expr}` internals are, because C doesn't
have Perl string interpolation. Today, per the memory notes, the XS codegen
for string interpolation is fragile (`lib/Chalk/Bootstrap/Perl/Target/EmitHelpers.pm`
has regex-level string-interpolation emitter).

**Recommendation.** For the near-term self-hosting milestone (validating
"Chalk parses `lib/` into something that can regenerate equivalent Perl"),
treat strings as opaque. Defer structural interpolation analysis to a
dedicated string-internals phase.

### Q5: FUNC vs LSTOP whitespace

**Grammar gap.** `print(1,2)+3` vs `print (1,2)+3` — Chalk treats them
identically; Perl treats the first as `print(1,2) + 3` and the second as
`print((1,2)+3)`.

**Usage in lib/.** No occurrences. Every `builtin (...)` site in lib/ is
either `return (...)` (a syntactic construct, not a named-function call) or
inside a comment. The specific pattern `BUILTIN (args) OP ...` does not
appear in real code.

**Does Chalk currently handle it?** The ambiguity does not manifest because
the pattern is absent.

**In scope for self-hosting?** NO. Not exercised by `lib/`.

### Q6: Array/hash slices

**Grammar gap.** `@arr[0,1,2]` and `@hash{'a','b'}` — Subscript rule admits
single Expression only, not ExpressionList.

**Usage in lib/.**

- Array slices with a SINGLE range Expression (not comma-separated list):
  - `lib/Chalk/Bootstrap/ConciseTree/Actions.pm:599` `_merge_trees(@nonempty[1 .. $#nonempty])`
  - `lib/Chalk/Bootstrap/ConciseTree/Actions.pm:613` `@block_ops = @block_ops[1 .. $#block_ops - 1]`
  - `lib/Chalk/Bootstrap/ConciseTree/Actions.pm:682` `_merge_trees(@nonempty[1 .. $#nonempty])`
  - `lib/Chalk/Bootstrap/ConciseTree/Actions.pm:690` `@block_ops = @block_ops[1 .. $#block_ops - 1]`
  - `lib/Chalk/Bootstrap/ConciseTree/Actions.pm:1406,1546` `my @body_trees = @trees[1 .. $#trees]`
  - `lib/Chalk/Bootstrap/Perl/Target/Perl.pm:672,690` `@args[0 .. $#args - 1]`

- Array slices with comma-separated list: 0 occurrences.
- Hash slices `@hash{...}`: 0 occurrences.
- Ref slices `%hashref{...}`: 0 occurrences.

**Does Chalk currently handle it?** YES for the forms used. The grammar's
`Subscript ::= Expression _ /\[/ _ Expression _ /\]/` rule accepts
`@arr[1 .. $#arr]` because the subscript is a single Expression (the `..`
operator builds a range). The `@` sigil on the LHS is handled by ArrayVariable.

The sigil-semantic issue (`@arr[...]` is a list-context slice while
`$arr[...]` is a scalar-context element) is orthogonal to grammar parsing
and is handled by the Perl runtime on the generated code.

**In scope for self-hosting?** NO. The comma-list subscript form (the actual
Q6 gap) is not used in `lib/`.

### Q7: Attribute argument syntax

**Grammar gap.** `:attr(args)` admits only a single QualifiedIdentifier as
the argument. Real Perl supports `:prototype($$)`, structured args, etc.

**Usage in lib/.**

Distinct attribute shapes used (direct `ag` extraction of the `:ident(arg)`
construct):

- `:isa(Chalk::IR::Node)`, `:isa(Chalk::Bootstrap::Target)`, etc. — all have
  a QualifiedIdentifier argument (possibly `::`-separated). Supported.
- `:isa($parent)` — appears once at `lib/Chalk/Bootstrap/Perl/Target/Perl.pm:341`,
  but this is inside a DOUBLE-QUOTED STRING: `$decl .= " :isa($parent)";`.
  It is not a Perl attribute in the source; it's the generated output. Not
  a grammar concern for `lib/`.
- `:param(recover)`, `:param(leo_enabled)`, `:param(start)`, `:param(returns)`
  — 4 occurrences in `lib/Chalk/Bootstrap/Earley.pm` and `lib/Chalk/IR/Graph.pm`.
  All arguments are bare identifiers (valid QualifiedIdentifier).
- `:param` (bare, no argument), `:reader` (bare) — 133+ and 132 occurrences.
  Supported by the grammar's first Attribute alternative.

**Does Chalk currently handle it?** YES for every attribute in `lib/`. The
grammar rule:

```
Attribute ::= /:/ _ QualifiedIdentifier
    | /:/ _ QualifiedIdentifier _ /\(/ _ QualifiedIdentifier _ /\)/ ;
```

covers all observed shapes: `:isa(Foo::Bar)`, `:param(name)`, `:reader`.

**In scope for self-hosting?** NO. Every attribute in `lib/` fits the
existing grammar. General Perl attributes (`:prototype($$)`, `:locked`, etc.)
are aspirational scope expansion.

### Q8: `\&foo` sub references

**Grammar gap.** Not in grammar.

**Usage in lib/.** Zero real occurrences. Two apparent hits:

- `lib/Chalk/Bootstrap/Perl/Target/C.pm:347,466` `$rewritten =~ s/\breturn\s*;/return \&PL_sv_undef;/g;`

Both are inside `s///` regex replacements — the `\&PL_sv_undef` is
replacement text being emitted as C output, not a Perl `\&sub` reference.

**Does Chalk currently handle it?** N/A — not exercised.

**In scope for self-hosting?** NO.

### Q9: `-X` file test operators

**Grammar gap.** Not in grammar.

**Usage in lib/.**

- `lib/Chalk/Bootstrap/DepChaser.pm:48,56,80,97`: `next unless -f $file;`
  and `next unless -f $path;` (4 occurrences).
- `lib/Chalk/Bootstrap/Runtime.pm`: `if ($ENV{CHALK_SO_PATH} && -f $ENV{CHALK_SO_PATH})`.

**Does Chalk currently handle it?** NO. The UnaryExpression rule only admits
`!`, `-` (numeric negation), `+`, `~`, `\`, and `not`. The `-f` form collides
with `-` as numeric negation: `-f $x` would parse as `- f $x`, which would
then be interpreted as unary minus on `f($x)` — and `f` is not a declared
sub, so TypeInference would reject it.

**In scope for self-hosting?** YES. `DepChaser.pm` is part of the bootstrap
toolchain; its `-f $file` guards are load-bearing for XS compilation.

Options:
- Add `-[efdrwxRWXoOzslpSbcugktTBAMC]` to the UnaryExpression rule.
- Rewrite `DepChaser.pm` and `Runtime.pm` to use `stat($file)` or
  `Cwd::realpath($file)` checks. Given this is ~5 sites, rewriting is a
  reasonable short-term workaround.

### Q10: Version strings in expressions

**Grammar gap.** `my $v = v1.2.3;` — supported in `use` but not in Atom
position.

**Usage in lib/.** Zero occurrences. The only `v\d+\.\d+` pattern is in
comments; `use 5.42.0;` is a bare version number (no `v` prefix), which the
grammar's Version rule `/v?[0-9]+(?:\.[0-9]+){2,}/` already accepts.

**In scope for self-hosting?** NO.

### Q11: `<...>` readline and `%` modulo

**Grammar gap.** Both are position-based ambiguities like Class 4.

**Readline `<...>` usage in lib/.**

- `lib/Chalk/Bootstrap/DepChaser.pm:169`: `my $source = <$fh>;` — ONE real site.
- `lib/Chalk/Bootstrap/ConciseOp.pm`: `"<$arity>  $name"` — string interpolation,
  not readline.
- Standard filehandles (`<STDIN>`, `<DATA>`, `<>`, etc.): 0 occurrences.

**Modulo `%` usage in lib/.**

- `lib/Chalk/Bootstrap/Earley.pm:... if ($pos % 10 == 0)` and `if ($pos % 1000 == 0)` — modulo on scalar.
- All other `%` usages are hash sigils (`%KEYWORDS`, `%BUILTIN_OPS`, etc.)
  in clearly non-binary-op positions (after `my`, `our`, `our`, at start of
  expressions).

**Does Chalk currently handle it?**

- Readline `<$fh>`: NOT in the grammar. `<` is the less-than BinaryOp; no
  production admits angle-bracket-delimited readline syntax.
- Modulo: YES. `%` is listed as a BinaryOp (`/[*\/%]/`). Grammar disambiguates
  modulo from hash sigil via context (BinaryExpression requires Expression
  on left and right). `$pos % 10 == 0` is unambiguous in practice.

**In scope for self-hosting?**

- Readline: PARTIAL. One site, in `DepChaser.pm` (bootstrap toolchain).
  Options: add a `Readline` atom rule (`/<\$[a-zA-Z_]\w*>/` | `/<[A-Z_]+>/`),
  OR rewrite the site to use `scalar <$fh>` with explicit `readline($fh)`
  or `File::Slurper`-style read.
- Modulo: NO. Used forms are unambiguous.

### Q20: Formats

**Grammar gap.** Not in grammar.

**Usage in lib/.** Zero occurrences. No `format NAME = ...` declarations,
no `write` calls.

**In scope for self-hosting?** NO.

## Recommended prioritization

### Must resolve for self-hosting

1. **Q9 (`-X` file tests)** — 5 sites in `DepChaser.pm` and `Runtime.pm`.
   Decision needed: grammar extension vs rewrite. Low-risk rewrite candidate:
   replace `-f $path` with `(-e $path && !-d $path)` no wait, those are also
   `-X`. Real alternatives: `stat($path)` and test the mode, or bring in a
   Path helper. **Recommend**: add `-f` (and `-d`, `-e` if convenient) to
   the UnaryExpression grammar rule. Strict scope, small grammar change.

2. **Q2 (`q(...)` / `qq(...)` delimiters)** — 3 sites in two files.
   **Recommend**: rewrite the 3 sites to use `q{...}` / `qq{...}` — zero
   grammar work, keeps the grammar narrow. Grammar extension would be
   simple (`/q\s*\(...\)/` patterns), but adding paren-delimited quote-like
   operators opens a class of tokens (`q!!`, `q//`, etc.) without principled
   reason to stop.

3. **Q11 readline** (1 site) — `DepChaser.pm:169` `my $source = <$fh>;`.
   **Recommend**: rewrite to `my $source = do { local $/; readline($fh) };`
   which replaces `<$fh>` with the explicit `readline` builtin. This also
   eliminates the readline/less-than ambiguity question entirely for this
   pass.

### Out of scope — defer indefinitely

- Q5 (FUNC vs LSTOP): not exercised.
- Q6 (comma-list slices): not exercised.
- Q7 (structured attribute args): not exercised.
- Q8 (`\&foo`): not exercised.
- Q10 (version strings in expressions): not exercised.
- Q11 modulo: used but unambiguous given existing BinaryOp grammar.
- Q20 (formats): not exercised.

These can be documented as "General Perl features NOT supported by the
Chalk grammar; not currently in scope". If Chalk were later extended to
compile arbitrary Perl, they would need addressing.

### Needs scope decision but not blocking

- **Q4 (string interpolation internals)**. Blocks XS/C backend for strings
  containing `${\ expr}`, but does NOT block Perl-backend self-hosting.
  Defer to the XS/C string-interpolation emitter design work.

## Key findings

### Grammar gaps NOT on the original question list

Discovered while auditing `lib/`:

1. **Punctuation variables (`$!`, `$@`, `$/`)**. The grammar's
   `ScalarVariable ::= /\$[a-zA-Z_]\w*/` rejects `$!`, `$@`, `$/`, and
   similar.
   - `lib/Chalk/Bootstrap/DepChaser.pm:167` uses `$!` inside a double-quoted
     string (opaque to grammar — no blocker).
   - `lib/Chalk/Bootstrap/DepChaser.pm:168` `local $/;` — `$/` as a
     declarator-scope variable. This IS a grammar gap: `local $/` cannot
     parse today.
   - `lib/Chalk/Bootstrap/DepChaser.pm:153` `... or die "Grammar eval failed: $@";`
     `$@` is inside a string — opaque — no blocker.
   - Recommendation: either rewrite `local $/;` (e.g. use `File::Slurper`)
     or extend `ScalarVariable` to include punctuation variables.

2. **`eval "string"`**. Chalk is documented to use `try/catch` only
   (per memory: `feedback_try_catch_not_eval`). But
   `lib/Chalk/Bootstrap/DepChaser.pm:153` has
   `eval "$generated; 1" or die "Grammar eval failed: $@";`. This is a
   documented exception case (eval-string on generated grammar code), but
   the grammar does not admit `eval EXPR` (there is no `eval` built-in
   rule). Must be rewritten or the grammar extended before `DepChaser.pm`
   can self-host.

3. **`open my $fh, ...`** — `DepChaser.pm:167`, `Earley.pm:... open my $sf, '<', '/proc/self/status'`.
   `open` is a regular builtin call and should parse as a CallExpression
   (unparenthesized). This likely works today (TypeInference scan-time
   validation with get_builtin), but worth verifying — `open` followed by
   a `my` declaration is a 2-expression list after a bareword builtin, which
   is the LSTOP form.

### Surprising non-usage

- **`tr///` / `y///`**: zero occurrences.
- **`\&sub` references**: zero real occurrences.
- **`-X` beyond `-f`**: zero occurrences.
- **Array slice with list subscript (`@arr[0,1,2]`)**: zero occurrences.
  This was one of the more worrying questions in the sweep, but `lib/` only
  uses the single-Expression range form that the grammar already accepts.
- **Formats**: zero occurrences.
- **Version strings in expression position**: zero occurrences.

The pattern: `lib/` uses a restricted, disciplined subset of Perl. Many of
the ambiguity questions identified by the toke.c sweep are aspirational for
a general Perl compiler, not load-bearing for self-hosting.

### Patterns where lib/ uses a restricted form

- **Array slices**: only `@arr[single_range_Expr]`, never comma-lists.
- **Attributes**: only `:isa(QI)`, `:param(ident)`, `:reader`, `:param`.
- **Quote-like operators**: `qw()` only for word lists; `q[]`/`q{}`/`qq[]`/`qq{}`
  for heredoc-style block strings; `qr//`, `m//`, `m{}`, `s///`, `s{}{}`
  for regex. No prototype-taking or unusual delimiters.
- **File tests**: only `-f`.
- **Modulo**: only on plain scalar numeric positions, never adjacent to hash
  sigil context.

## Questions for the maintainer

1. **Q9 (`-X` file tests)**: prefer grammar extension (admit `-f` etc. as a
   unary operator) or rewrite the 5 sites in `DepChaser.pm`/`Runtime.pm`
   using `stat()`? The grammar extension is conceptually cleaner (these
   are real Perl operators); the rewrite keeps the grammar narrow.

2. **Q2 (paren-delimited quote-like operators)**: rewrite 3 sites to `{}`
   or extend grammar to admit `q(...)` / `qq(...)`? Recommend rewrite —
   grammar extension would re-open the delimiter-admission question.

3. **Q11 readline**: one site in `DepChaser.pm`. Rewrite to explicit
   `readline($fh)`, or add grammar rule for `<$fh>`?

4. **Discovered gap: `local $/;` and `eval "..."`** in `DepChaser.pm`.
   Is `DepChaser.pm` itself in the self-hosting scope? If yes, both sites
   need treatment (rewrite or grammar extension). If `DepChaser.pm` is
   tooling that runs under stock Perl (not under Chalk), this is not a
   self-hosting blocker.

5. **Q4 (string interpolation internals)**: confirm scope. Perl-target
   self-hosting does not need this. XS/C-target does. Is XS/C target in
   the self-hosting acceptance criteria, or only the Perl target?
