### `Target::Perl::generate($mop)` is a façade over the legacy path

`lib/Chalk/Bootstrap/Perl/Target/Perl.pm` lines 75-194:

```
method generate($input) {
    if ($input isa Chalk::MOP) {
        return $self->_generate_from_mop($input);
    }
    # legacy: ClassInfo::Program path
    return $self->_emit_program($input);
}

method _generate_from_mop($mop) {
    # For each class:
    #   For each method:
    #     my $body_ref = $method->body;
    #     my @method_body = (ref($body_ref) eq 'ARRAY' && $body_ref->@*)
    #         ? $body_ref->@*
    #         : $self->_body_from_graph($method->graph);
    #     push @body, Chalk::IR::MethodInfo->new(body => \@method_body, ...);
    #   ...
    #   $self->_emit_class_decl($class_info);
}
```

The MOP path:

1. Reads `$method->body` (arrayref stored on `MOP::Method`,
   populated by Actions).
2. Falls back to `_body_from_graph($method->graph)` only when body
   is empty.
3. Synthesizes `Chalk::IR::MethodInfo` / `Chalk::IR::ClassInfo` /
   `Chalk::IR::SubInfo` wrappers from the MOP data.
4. Hands the synthesized wrappers to the same legacy `_emit_*_decl`
   helpers, which then `->body()`-walk them.

So Phase 4 *contract* is met (`generate($mop)` exists, returns
HashRef[Str], no backchannel in signatures) but Phase 4 *exit
criterion* ("No `->body()` readers remain in `lib/`") is not. The
MOP path is a translation layer around the legacy reads.

### The control-chain is not threaded for all side-effect statements

The Phase 3a-migration spec says:

> Each side-effect action (`VariableDeclaration`, `Assign`, `Call`,
> etc.) reads `$ctx->scope` for variable resolution, constructs its
> typed node, merges it into `$ctx->graph`, and extends the Context
> with the updated graph and scope. … Each side-effect node's
> control input is the previous side-effect node (or `start` for
> the first).

This is true for `VarDecl` and `Return` but **not** for `Call` as a
bare statement.

#### Probe 1 — `push @list, X` as a bare statement

Source:

```perl
class Probe {
    method m() {
        my @list = (1, 2);
        push @list, 3;
        return scalar @list;
    }
}
```

State after parse:

- `$method->body`: 3 items in source order — `VarDecl`, `Call`, `Return`.
- `$method->graph->nodes`: 9 nodes — `Start`, `Constant×3`,
  `HashRef`, `Constant`, `Call`, `Return`, `VarDecl`, `Constant`.
- `Call->inputs[0]`: `Constant`, NOT a control node.
- Control chain `Return->inputs[0]` → `VarDecl->inputs[0]` →
  `Start`. The `Call` is not on this chain.
- `_body_from_graph` (the existing control-chain walker) recovers
  only 2 nodes: `VarDecl` and `Return`. The `Call` is missed.

**The Call is in the graph but off the control chain.** It is
reachable via data flow (the `@list` reference inside the Return ↔
VarDecl edge), but not via control inputs.

#### Probe 2 — `print "hi"` as a bare statement

Source:

```perl
class P2 {
    method m() {
        my $x = 1;
        print "hi";
        return $x;
    }
}
```

State after parse:

- `$method->body`: 3 items in source order — `VarDecl`, `Call`, `Return`.
- `$method->graph->nodes`: 5 nodes — `Start`, `Constant×2`,
  `VarDecl`, `Return`. **No `Call` node at all.**

`print "hi"` is a pure side effect with no return value being
consumed. Without a data-flow consumer, the Call node was either
never built, or was built and never merged into the graph. Either
way, the statement exists only in `$method->body`. Remove the body
field and `print` disappears from generated code.

#### What this means for Phase 4

The codegen `->body()` reads cannot be replaced with graph walks
until two distinct problems are solved:

1. **Source order from a graph.** Graph nodes are topologically
   ordered, not source-ordered. A control-chain walk recovers
   source order, but only for nodes on the control chain.
2. **All side-effect statements on the control chain.** The
   semantic actions for `Call`-as-statement, `Assign`-as-statement,
   and similar bare-side-effect statements need to thread their
   nodes onto the control chain when they appear at statement
   position. Today only `VarDecl` and `Return` do this.

Problem (2) is the real blocker. It is Phase 3a-migration's
unfinished scope. The plan's wording ("Call, etc.") implied this
was covered; the implementation only covered VarDecl/Return.

### The Phase 3c reachability test is under-scoped

`t/bootstrap/mop/build-graph-reachability.t:60-80` —
`check_reachability($method, $label)` — walks ANY-inputs BFS from
Return nodes and asserts that every `VarDecl` is reached. It does
NOT assert that every `Call` (or `Assign`, etc.) is reached.

This is why Phase 3c's "every statement reachable" exit criterion
shows as passing today even though `print "hi"` produces no Call in
the graph at all. The test was scoped to VarDecls (the only
statement Phase 3a-migration actually threaded), not to "every
statement."

**The Phase 3c exit criterion was claimed but not fully verified.**

## Recommended next step

**Finish Phase 3a-migration's actual scope.** Thread every
side-effect statement (`Call`, `Assign`, and any others that appear
at statement position) onto the control chain. Specifically:

1. Identify every semantic action in `Actions.pm` that produces a
   side-effect IR node when its source position is a bare statement.
2. For each, mirror the VarDecl/Return pattern: read
   `$ctx->scope->control` as `control_in`, construct the node with
   that as `control_in`, extend the scope with
   `$scope->with_control($new_node)`.
3. Strengthen `build-graph-reachability.t` to assert every Call,
   Assign, and other side-effect statement is reachable on the
   control chain, not just VarDecls. The test as written cannot
   catch the gap this audit found.
4. Once `_body_from_graph` recovers full source-ordered bodies for
   the green-eval file set, replace the 18 `->body()` reader sites
   with graph walks. This becomes mechanical at that point.

The work is genuinely Phase 3a, not Phase 4. The Phase 4 contract
is already in place; what's missing is the upstream data the
contract was supposed to read.

### Why not just expand `MOP::Method->body` and use it?

It's tempting — `body` already exists on the metaobject and codegen
could read it directly without going through MethodInfo. That
collapses the `_generate_from_mop` façade and removes ~110 lines of
synthesis code from `Target/Perl.pm` alone. The 18 `->body()` reader
sites become `$mop_method->body()` reads.

But this trades one form of debt for another:

- The Phase 4 stated goal ("codegen walks the graph") becomes
  unmet, intentionally.
- `MOP::Method->body` becomes a parallel-state field alongside the
  graph, forever — exactly the kind of legacy we were trying to
  delete.
- The body arrayref is what Phase 3a-migration was supposed to make
  unnecessary by threading the control chain. Promoting it to a
  durable MOP field is a regression in design intent.

Worth keeping as a fallback if completing the control-chain
threading turns out to be too large for one session, but not the
default.
